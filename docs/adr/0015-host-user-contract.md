# Hosts and users live in separate repos, bound by a shared contract

**Status:** Accepted. Mechanic 7's *curated-catalog* framing (a restricted `evalModules` over re-exposed system options) is **superseded by [ADR-0018](0018-user-confinement-manifest-greeter.md)** (the `contract.requests` channel); every other mechanic stands.

The goal is that **any host can enable a user and, on rebuild, that user transparently works** — login, dotfiles, the features they need — and that **a host can deny features a user introduces**. Today `users/` lives inside this host repo and is coupled to it in five ways (`self.lib.getSecret*`, `self.homeManagerModules.*`, host flake inputs, the `custom.users` schema pulled in by the `base` profile, and secrets living in the host's `stash`). "Enable a user on an arbitrary host" cannot hold while the user *is* part of one host repo.

The decision is **three repos joined by a shared contract**, not a self-contained user flake:

- A small **contract** flake declares the schemas both sides agree on: the identity option set (today `users/identity-options.nix`), the home-profile meta options, and the **feature/capability vocabulary** that denial keys on. Both host repos and user repos depend on it. A self-contained user flake was rejected because "deny a feature" is fundamentally a *negotiated interface* — it needs a shared vocabulary of features that both sides name, which only a third agreed-upon schema can provide. In practice the contract is a **kit**, not just schemas: it also ships the host-invariant realization (mechanic 5), the recipients-from-grants derivation, and a **conformance suite** proving the patterns across synthetic hosts × users (the host supplies only the `platform` *binding*). See `CONTEXT.md` and [0019](0019-feature-configuration-aggregates.md).
- A **user** is a thin module written against that contract — public identity + home config + its own secrets + the features it *offers*. It is host-agnostic: it never names a secrets backend or a host's `self`/inputs.
- A **host** consumes the contract, materialises the system account from the user's public identity, **grants** the features it chooses, and injects the runtime platform.

Seven mechanics make this work, each chosen to preserve a property the fleet already relies on:

1. **Secrets follow features; public identity travels with the user.** "User secret" is three tiers: public identity (`authorizedKeys`, name, email, username, profile) is *not secret* and lives as plaintext in the user repo; `hashedPassword` is a one-way hash whose handling depends on the user repo's visibility (see *Threat model & hardening*) — plaintext config in a *private* repo (keeping enable crypto-free), but sops-encrypted and per-host re-keyed if the repo is public/shared; only **feature secrets** (private keys, restic, API tokens) are real secrets, and a host learns one **only when it grants the feature that consumes it**. Baseline "enable a user" is therefore **crypto-free**. This extends ADR-0003 to feature granularity: the user's admin/master key **never reaches any host** — it stays on the trusted machine and is what *re-keys* a host into one specific feature secret on grant. A headless or code-executing host that grants only a login shell holds **no private key material** — there is nothing to leak.

2. **Default-closed grant.** A host runs only what it explicitly grants (`custom.users.<user>.granted.<feature>`); "deny" is the absence of a grant. The user's phrasing ("hosts deny features a user introduces") suggested default-*open* veto, but default-closed was chosen: it is host-sovereign by construction, has no fleet-wide blast radius (a new user feature cannot silently activate everywhere), and makes the per-feature secret model safe — no grant ⇒ no re-key ⇒ no secret.

3. **A feature token-gates a user-owned bundle** (`mkIf granted.<feature>`), mirroring ADR-0013 — the fleet's one mental model of bundle + enable-flags + disabled-is-a-no-op, now applied to the host↔user boundary instead of a fourth pattern. The escape hatch is ADR-0014: features touching version-divergent home-manager options must be conditionally *imported*, not just gated, because `mkIf` cannot suppress unknown-option errors across home-manager version skew.

4. **Packages ride features; overlays `follows` the host's nixpkgs.** A feature that needs an overlay or custom package carries it; the user flake brings the overlay as its own input but pins `inputs.nixpkgs.follows` to the host's, so there is **one** nixpkgs (no divergence, no double instantiation). Granting accepts the packages; denying keeps them out.

5. **The contract realizes the account; grants decide its powers.** The contract ships a host-invariant module that maps the identity schema to `users.users` (`isNormalUser`, `hashedPassword`, `authorizedKeys`, `description`) — one realization, so no per-host mapping boilerplate to drift. But *privileged* powers are deliberately **not** realized from raw user-declared data: display-manager/networkmanager enablement and privileged group membership are side-effects of *granted features*, host-gated, not of a user merely declaring `profile = "gui"` or an `extraGroups` list. Trusting the user's group list verbatim across the host↔user trust boundary would let a user repo add itself to `docker`/`libvirtd` on any host that enabled it — a host compromise. So today's `users/identity.nix` splits: the mechanical account → the contract realization module; the powers → the features that grant them.

6. **The injected platform is minimal and contract-typed.** What keeps a user module from naming the host directly is a `platform` attrset the host passes via `specialArgs` — but it carries *only* what the contract declares an interface for, which is realistically just a **secrets accessor**. Everything the old `self.*` coupling carried routes elsewhere: grants and host facts through `config`/`osConfig`, overlays/packages through the user's own `follows`-pinned inputs, and host-invariant shared modules (today `self.homeManagerModules.*`) through the contract — *not* the runtime channel, since a static module passed through `specialArgs` is a category error. A rich convenience bundle was rejected: `specialArgs` are not type-checked the way options are, so an undeclared platform surface becomes a second, unwritten contract whose missing field is a late `attribute missing` error rather than an option-system one. The secrets accessor is **declare-and-locate, returning a runtime path — never an eval-time value**: a feature calls `platform.secret "restic-password"`; the host's implementation registers it with *its* sops (`sops.secrets."<user>/…"`, owner/mode) so the admin's grant-time re-key provisions it, and hands back a path decrypted at *activation*. Plaintext never touches eval (no IFD, no secret in the store), and an ungranted feature simply never calls `platform.secret`, so nothing is registered or re-keyed. The admin re-key action itself is **build-time tooling, not a module arg** — it never enters `specialArgs`. **In-repo realization (slice 03):** the platform is implemented as a typed `custom.platform` *option* set (resolvers of type `functionTo path`) the host binds, rather than a `specialArgs` attrset — which is what the type-checking argument above actually asks for, since a host that forgets to bind it gets an option error, not a late `attribute missing`. The first resolver abstracts the secret *source* (the encrypted sops file); the decrypted value is still located at activation via standard `config.sops.secrets.<n>.path`. The exposed-host rule is enforced as a NixOS `assertion` keyed on `custom.host.exposed` and the contract's per-feature `secretBearing` metadata.

7. **The host↔user boundary is a *type* boundary, not a trust boundary (target).** A user that exports arbitrary modules can set *any* option (`users.users.root`, `security.sudo`, an activation script) outside every grant gate — so with raw modules "deny" is cosmetic and enabling a user means trusting that repo as root. The target is therefore that the contract evaluates the user's modules against a **restricted option universe** (`lib.evalModules` whose `options` are *only* the feature vocabulary) and translates the result into the real NixOS/home config the host applies. The user keeps module expressiveness (`mkIf`, `lib`, their own `imports`) but can set only options that *exist* in that universe — `users.users.root` is unsettable by construction, so deny is enforced by the type system, not by review. The cost is that the contract becomes a **curated capability catalog**: re-exposing e.g. `programs.git` as a `features.git` option set and translating it, with anything outside the catalog unavailable until added — which re-litigates ADR-0014's home-manager version skew at the catalog edge. This confinement is **deferred during the in-repo migration**: while host and user share one trust domain the user is imported as ordinary modules (the trusted "model A" posture), and the restricted `evalModules` boundary is built only at the repo-split, when an external repo first crosses the trust line.

## Consequences

- "Enable a user" and "grant a feature" are **two distinct acts**. The first is a crypto-free import of public config. The second (`grant feature F on host H`) is a re-key of H into F's secret *plus* flipping the grant flag — exactly the ADR-0002 re-key step, now per-feature. Tooling must make the re-key explicit, or a granted feature fails at sops activation.
- Secret access must sit **inside** a feature's `mkIf`, so an ungranted feature never forces its secret. `warnMock` (ADR-0012) already degrades a missing secret to a loud warning, not a crash, so denial fails gracefully.
- The **contract flake is a dependency of both repos**, so a schema change is breaking across the boundary — it must be versioned/pinned like any other flake input, and identity/feature options can no longer be edited in lockstep with their consumers.
- A user repo must pin `nixpkgs.follows` to the consuming host (the host sets `inputs.user.inputs.nixpkgs.follows = "nixpkgs"`); forgetting this resurrects a second nixpkgs.
- The account realization lives in the contract, but the `extraGroups` identity field can no longer be honoured wholesale: privileged group membership accrues from *granted features*, not from a user-declared list, or a user repo could self-escalate on any host that enables it. Auditing what the separate-repos split does to each currently-trusted identity field is part of the migration.
- The `platform` interface must be declared in the contract before a host injects it or a user consumes it; adding host-provided runtime capability is a *contract* change, not an ad-hoc `specialArgs` field. The host wires the same `platform` into both the system evaluation (`specialArgs`) and home-manager (`extraSpecialArgs`).
- This is the **target**, not the current state. The migration starts from the already-extracted `users/identity-options.nix` (the contract's first artifact) and the existing `custom.users` schema, then lifts the secrets-accessor behind an injected interface and reframes home profiles as offered features.

## Threat model & hardening

The boundary's strength rests on a few rules beyond the mechanics above:

- **Treat all user-declared data as untrusted input.** The `extraGroups` / `gui→sddm` escalation (mechanic 5) is one instance of a class: any field the realization reads that confers host-side power is an escalation vector. The contract must clamp every privilege-bearing field; powers come only from host grants, never from user-declared values.
- **Revocation is *remove recipient + rotate*, not just ungrant.** A host ever granted a secret-bearing feature has seen the cleartext (in `/run/secrets`, possibly in backups); un-granting must remove the host as a sops recipient *and* rotate the secret. Automate it (e.g. `just revoke <host> <feature>`) so the rotate can't be forgotten.
- **Derive sops recipients from the grant declarations.** One source of truth: re-key tooling reads a host's grant set and re-keys exactly those feature secrets — making "re-key the wrong host" structurally impossible and keeping recipients ≡ grants.
- **Exposed hosts may not grant secret-bearing features.** Codify ADR-0003's instinct as a contract assertion: an agent/headless host class (e.g. `kelpy`, which runs a Claude Code agent as `inkpotmonkey`) is login-only, and granting a secret-bearing feature there is an eval error. The hosts most likely to be compromised then hold no cleartext at all — the strongest practical mitigation for the inherent runtime-cleartext exposure.
- **Supply chain: pin user and overlay inputs by commit hash and review updates.** `nixpkgs.follows` pins the base but not an overlay's own code, which runs at eval/build on the host's builder. Trust in a user repo (model A in-repo) and its overlays must be explicit, not laundered through `follows`.
- **`hashedPassword` by repo visibility.** Private user repo → plaintext hash is fine and "enable" stays crypto-free. Public/shared user repo → sops the hash (use a costly KDF such as yescrypt) and accept that every granting host re-keys this one secret even for a bare login. *Crypto-free enable* and *no-crackable-hash-in-the-open* are in genuine tension; pick by visibility.

## Considered Options

- **User as a self-contained flake** — rejected: "deny a feature" requires a negotiated, shared feature vocabulary; a self-contained flake gives the host no agreed names to deny.
- **Default-open veto** (user features on by default, host vetoes) — rejected: fleet-wide blast radius (a new feature auto-activates everywhere until each host vetoes) and it breaks the per-feature secret safety that default-closed gives for free.
- **Feature as a module the host imports on grant** — mostly rejected: it shatters the user's home config into N importable fragments and adds a fourth composition pattern; token-gating (ADR-0013) is the default, with conditional import (ADR-0014) only where version skew forces it.
- **User owns all package inputs freely** — rejected: risks a second nixpkgs (divergence, double-eval, cache misses) and removes the host's say over what packages arrive, cutting against "host can deny."
- **Host provides pkgs, user consumes only** — rejected: the user could use only packages the host already has, breaking "transparently works on any host" the moment a user wants something new.
- **Each host owns the `custom.users → users.users` mapping** — rejected: it duplicates identical realization boilerplate across host repos (the drift we just removed by sharing `identity-options.nix`). Centralise the mechanical realization in the contract; keep only the *policy* (group grants, display-manager enablement) host-side, expressed as feature side-effects.
- **A rich `specialArgs` platform** (lib helpers, host facts, shared modules, a re-key handle) — rejected: `specialArgs` aren't type-checked, so it becomes a second unwritten contract; everything it would carry already routes through `config`, the contract, or the user's own inputs. Keep the platform to the contract-declared minimum.
- **User exports arbitrary modules permanently** (model A as the end state) — rejected *as the target*: a module can set any option outside any grant, so "deny" is unenforceable and enabling a user trusts the repo as root. Retained only as the *in-repo migration* posture, where host and user share one trust domain.
- **User exports flat data only** (model B) — rejected: it makes deny enforceable but collapses the user's config power to literal file payloads (no `mkIf`, no `programs.*` logic). The restricted-`evalModules` target (model C) gets enforceable deny *and* keeps module expressiveness.
