# Daemon-restricted users and host-built package profiles: the `nix-daemon` feature, package policy, and graceful degradation

**Status:** Proposed. Related to [ADR-0032](0032-prebuilt-binding-mode.md) (pre-built
binding mode); requires the `contractPackage`'s `packages` manifest field defined there.

## Context

The pre-built binding mode (ADR-0032) establishes that packages in a user's home are
**advisory**: the user owns their package versions and the host owns the feature grants.
This ADR addresses the complementary question — what happens when a host wants to restrict
which programs appear in a user's session?

The key insight that frames the answer: **packages are only advisory when the user has Nix
daemon access.** A user with the daemon socket can run `nix shell nixpkgs#<anything>` to
install and execute arbitrary software, bypassing any home-config-level policy. Hard program
restriction therefore requires removing daemon access at the OS level, not configuring it at
the home-manager level. Once daemon access is removed, the programs that appear in the store
are bounded to what the host placed there when activating the user's home.

The threat being closed by this ADR: **a user running software that was not in their home
config at all.** This is distinct from (and easier to close than) restricting transitive
dependencies or managing individual package versions.

## Decision

### 1. New feature: `nix-daemon`

A new entry in `features.nix`:

```nix
nix-daemon = {
  grant = "access to the Nix daemon for this user (host grant)";
  groups = [ "nix-users" ];
};
```

The host configures `nix.settings.allowed-users = ["@nix-users"]`. A user granted
`nix-daemon` is in the `nix-users` group and can use the daemon freely. A user denied it
cannot add new store paths, build derivations, or install packages beyond what the host
places in the store at activation time. This is a **privileged group grant** (excluded from
the safe set by the group check, not by exec-payload), so it is build-time-only — the
greeter never auto-grants daemon access.

### 2. Package policy: the inclusion list

A new host-level option surface:

```nix
custom.host.packagePolicy.allowedPrograms = [ "firefox" "vim" "git" "curl" ];
```

Each entry resolves to `pkgs.${name}` from the host's nixpkgs pin. One canonical version
per program — the most recent version in the host's nixpkgs. No multi-pin version
negotiation; the host's nixpkgs update cadence determines how current the programs are.

The effective package profile for a user is the **intersection** of the inclusion list and
the user's package manifest (the `packages` field in `contract-requests.json`): programs the
user declared and the host allows. Programs the user declared but the host does not allow are
absent; programs the host allows but the user did not declare are not imposed.

### 3. Host-built profile replaces `~/.nix-profile` at activation

After running `$contractPackage/activate` (which sets up dotfiles and the user's own package
profile), a privileged activation step:

1. Reads `contract-requests.json` to get the user's package manifest
2. Intersects the manifest with `allowedPrograms`
3. Builds a Nix profile containing the approved programs from `pkgs`
4. Replaces `~/.nix-profile` (and `~/.nix-profile/bin` on `$PATH`) with the host-built profile

The user's session then has access to: their full home config (dotfiles, rc files, configs
generated by `programs.*`) **and** only the host-approved programs in `$PATH`.

### 4. Non-compliant homes always deploy — graceful degradation

A home declaring programs not on the inclusion list is **not rejected**. It deploys as-is;
the unapproved programs are simply absent from the session profile. Home-manager configs
generated for unapproved programs (e.g. `~/.config/nvim/` from `programs.vim.enable = true`
when vim is off the list) are present but inert — the config file exists, the binary does not.

This is the correct behaviour: the user's expressed configuration is respected as fully as
the host's policy allows. Rejection is unnecessarily harsh and could leave users without a
working session.

### 5. The residual risk is accepted and documented

A technically capable daemon-restricted user can execute programs by their absolute store
path (`/nix/store/<hash>-<name>/bin/<program>`) without the binary appearing in `$PATH`.
The programs reachable this way are bounded to the user's own activation package's transitive
closure — software the host audited and accepted when it updated its `contractPackage` pin.

Closing this residual requires OS-level mechanisms (Landlock, mount namespaces) with
complexity equivalent to lightweight containers. This is out of scope. The threat being
closed by this ADR (software not in the home config at all) is fully closed by daemon
restriction + activation. The residual (store-path access to transitive dependencies) is
accepted.

### 6. Upgrade path to hard enforcement

For hosts that need hard enforcement — where even the residual risk is unacceptable — the
exit is the **inline-eval mode** (ADR-0032): the host evaluates the home against a filtered
`pkgs` that only exposes allowed programs. Programs not in the allowed set fail to build.
`programs.*` still works; the restriction lives in `pkgs`, not the home module. This
abandons the pre-built mode for those users. It is documented as Tier 3 and deferred.

## Why these decisions

- **`nix-daemon` as a feature** fits the existing grant/deny model cleanly. The host makes a
  deliberate decision (grant or deny daemon access) and the realization enforces it via group
  membership. No new mechanism — the groups path already exists.
- **Inclusion list rather than allow-list assertions** avoids rejecting users. Graceful
  degradation ("you get what's allowed") is a better user experience than "your session
  doesn't start."
- **Most-recent-version matching** avoids multi-pin complexity. The host carries one nixpkgs
  pin; users get whatever version is in it. The host's update cadence is the version policy.
- **Profile replacement rather than activation filtering** is the simplest mechanism: the
  activation runs unmodified (preserving all config), then the host asserts its package policy
  in one step. No interception of the activation package internals is needed.

## Consequences

- `features.nix` gains `nix-daemon`. `realization.nix` handles the `nix-users` group
  membership exactly as it handles other granted groups. No new realization logic.
- A new host option surface: `custom.host.packagePolicy.allowedPrograms`. This is a host
  **binding** (the contract ships the mechanism; the host supplies the list), consistent with
  the platform/display binding pattern (ADR-0024).
- The conformance suite gains a VM test (`daemon-restricted-vm.nix`):
  - Host denies `nix-daemon`; `allowedPrograms` contains `hello`
  - `contractPackage` declares `hello` and `curl` in its manifest
  - After activation: `su -c "hello" testuser` exits 0; `su -c "nix shell nixpkgs#curl" testuser`
    exits non-zero (no daemon); `curl` is absent from PATH
- The CONTEXT.md terms `program scope`, `nix-daemon feature`, `daemon-restricted user`, and
  `package policy / inclusion list` are stable vocabulary as of this ADR.
